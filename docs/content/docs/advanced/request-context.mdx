Use `Visit` in an Axum handler when work depends on the current Inertia visit. Use
`ShareContext` inside a `Share` provider when application-wide props need borrowed
request data. `RequestContext` is the lower-level, framework-neutral header model.

## Prerequisites

No Cargo feature is required for these types. The typed `Visit::selects` example
below uses `#[derive(InertiaPage)]`, which requires the default `macros` feature.
Install the application with `RouterInertiaExt::inertia` so the provider, handler,
and finalizer use the same parsed visit.

## Skip unselected eager work

`Visit::selects` is an escape hatch for ordinary eager data that must be loaded
before a page value can be constructed. It accepts the component-qualified
`PropKey` generated by `InertiaPage`:

```rust
use inertia_axum::{InertiaPage, RouterInertiaExt as _, Visit};
use serde::Serialize;

#[derive(Serialize)]
struct AuditRow {
    id: u64,
}

#[derive(InertiaPage)]
#[inertia(component = "Reports/Index")]
struct ReportsPage {
    title: String,
    audit: Vec<AuditRow>,
}

async fn index(visit: Visit) -> ReportsPage {
    let audit = if visit.selects(ReportsPage::AUDIT) {
        load_audit().await
    } else {
        Vec::new()
    };

    ReportsPage {
        title: "Reports".to_owned(),
        audit,
    }
}

async fn load_audit() -> Vec<AuditRow> {
    vec![]
}
```

On a matching `GET` partial reload that omits `audit`, the placeholder is filtered
out and the database work is skipped. `selects` returns `true` for full visits,
non-`GET` requests, component mismatches, and matching partial reloads that select
the key. If both `only` and `except` are present, exclusion wins.

Do not use `selects` around `lazy`, `optional`, `defer`, or `once` resolver closures.
The finalizer applies their complete policy set before it invokes a selected
closure, so an unselected future is never constructed or polled. `selects` is only
an eager-work optimization; it does not replace the finalizer's selection rules.

## Inspect the visit

`Visit` is an infallible Axum parts extractor. The Inertia layer stores an owned
clone in request extensions before the handler runs; without that extension, the
extractor reconstructs the visit from the request parts.

| Accessor | Meaning |
| --- | --- |
| `is_inertia()` | The request contains `X-Inertia`. The header value is not validated. |
| `is_partial()` | An `X-Inertia-Partial-Component` header is present. This alone does not prove that it matches the response component. |
| `is_prefetch()` | `Purpose` is exactly `prefetch`, compared case-insensitively. |
| `is_reload()` | `Cache-Control` contains a case-insensitive `no-cache` token. |
| `version()` | The raw client asset version, if present. |
| `requested_props()` | Borrowed `X-Inertia-Partial-Data` names. |
| `excluded_props()` | Borrowed `X-Inertia-Partial-Except` names. |
| `reset_props()` | Borrowed `X-Inertia-Reset` names. |
| `except_once_props()` | Borrowed once-cache keys already held by the client. |
| `error_bag()` | The requested validation error-bag name. |
| `selects(key)` | Whether an ordinary eager typed prop is selected for this method and component. |

The iterators avoid allocating vectors. Reset names do not omit values; they tell
the response metadata step to remove merge and scroll behavior for those props.
Once exclusions contain cache keys, which can differ from prop names. Prefetch and
reload are observations only—the adapter does not change prop selection merely
because either flag is set.

`Visit` intentionally does not expose arbitrary headers, extensions, the request
method, or URI. Combine it with ordinary Axum extractors when a route needs those
values.

## Prepare shared data from request state

`ShareContext` exists only during the synchronous `Share::share` call. It borrows
the current method, URI, headers, parsed `Visit`, and extensions already visible to
the Inertia layer:

```rust
use inertia_axum::prelude::*;
use std::convert::Infallible;

#[derive(Clone)]
struct CurrentUser {
    name: String,
}

#[derive(InertiaProps)]
struct AppShared {
    user_name: Prop<String>,
}

#[derive(Clone)]
struct AppShare;

impl Share for AppShare {
    type Props = AppShared;
    type Error = Infallible;

    fn share(&self, context: ShareContext<'_>) -> Result<Self::Props, Self::Error> {
        let name = context
            .extension::<CurrentUser>()
            .map_or_else(|| "Guest".to_owned(), |user| user.name.clone());

        Ok(AppShared {
            user_name: lazy(move || async move {
                Ok::<_, Infallible>(name)
            }),
        })
    }
}
```

`extension::<T>()` returns `&T`; it neither clones the extension nor exposes the
whole extension map. Clone or otherwise project the small owned value needed by an
asynchronous `Prop`, because resolver closures are `Send + 'static` and cannot
retain a `ShareContext` borrow.

An extension must be inserted before the Inertia service handles the request. This
ordering, reduced from the typed shared-data integration test, makes `CurrentUser`
visible during shared preparation:

```rust
use axum::{Extension, Router};

let router = Router::new()
    // Add routes here.
    .inertia(InertiaApp::default_root().share(AppShare).build()?)
    .layer(Extension(CurrentUser { name: "Ada".to_owned() }));
# Ok::<_, inertia_axum::ConfigError>(router)
```

The provider is prepared before the route handler on requests that pass the
pre-handler asset-version check, including ordinary non-page routes. A stale
Inertia `GET` can return its version-conflict response before shared preparation.
Returned shared props are resolved only if the handler ultimately returns an
Inertia page. Keep synchronous preparation small; return owned `Prop` resolvers for
asynchronous work. Only one provider is installed, and a later `.share(...)` call
replaces the earlier one.

## Parse the lower-level context

`RequestContext` owns the parsed protocol-header state and is useful to framework
integrations and focused tests:

```rust
use axum::http::HeaderMap;
use inertia_axum::RequestContext;

fn inertia_context(headers: &HeaderMap) -> RequestContext {
    RequestContext::from_header_fn(|name| {
        headers.get(name).and_then(|value| value.to_str().ok())
    })
}
```

It exposes the same visit flags plus `partial_component`, the raw infinite-scroll
merge intent, allocating and iterator forms of header lists,
`partial_reload_matches`, `without_partial_reload`, and `filter_props`. The
`without_partial_reload` copy clears component, include, exclude, reset, and scroll
intent state but preserves once exclusions. `filter_props` is primarily for
framework adapters: it leaves non-object values unchanged and always preserves an
`errors` object in object props.

Axum applications normally use `Visit` and `ShareContext`. The compatibility
`InertiaRequest` extractor also exposes `context()`. The advanced module re-exports
the same type as `InertiaRequestContext`; the alias does not add behavior.

## Trust and ownership limits

- Protocol headers are client-controlled. Do not treat `is_inertia`, prefetch,
  reload, error-bag, or prop-selection values as authentication or authorization.
- A raw partial header list is not the effective selection. Partial filtering
  requires an Inertia `GET` and a component match; `except` then takes precedence.
- `RequestContext` contains protocol headers only. It does not retain a request
  body, method, URI, referer, or arbitrary Axum extensions.
- `ShareContext` is borrowed and read-only. Project owned values during `share`;
  do not try to retain the context for later resolution.
- Use the iterator accessors on hot paths. The `Vec<String>` accessors intentionally
  allocate compatibility copies.

See [Request headers](/docs/reference/request-headers) for the complete wire map,
[Partial reloads](/docs/guides/partial-reloads) for client selection, and
[Shared data](/docs/concepts/shared-data) for merge and shadowing behavior.
