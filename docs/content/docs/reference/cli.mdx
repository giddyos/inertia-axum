`cargo-inertia` is optional project tooling. It is a separate binary crate and is not built by the workspace's default-member commands.

From a repository checkout, install it into Cargo's binary directory:

```sh
cargo install --path crates/cargo-inertia
```

The executable accepts both `cargo inertia ...` and direct `cargo-inertia ...` invocation.

## Command summary

```text
cargo inertia init [--framework <svelte|react|vue>] [--package-manager <auto|npm|pnpm|bun>] [--ssr <none|managed-node|external>] [--yes]
cargo inertia dev [--frontend <directory>] [--package-manager <auto|npm|pnpm|bun>] [--host <host>] [--port <port>] [-- <cargo args>]
cargo inertia check [--path <root>] [--frontend <directory>] [--built] [--ssr]
cargo inertia sync [OUTPUT] [-p <package>] [--workspace] [--layout <auto|single|modules>] [--check] [--clean]
```

Errors are written to stderr with a `cargo inertia:` prefix and a non-zero exit code.

## `init`

`init` creates a minimal Vite frontend at `<path>/frontend`. `--frontend` is required; `--path` defaults to the current directory.

| Framework | Inertia adapter | Page file |
| --- | --- | --- |
| Svelte | `@inertiajs/svelte` `3.6.1`, Svelte `5.55.7`, Vite `8.1.4` | `src/Pages/Home.svelte` |
| React | `@inertiajs/react` `3.6.1`, React `19.2.4`, Vite `8.1.4` | `src/Pages/Home.tsx` |
| Vue | `@inertiajs/vue3` `3.6.1`, Vue `3.5.29`, Vite `8.1.4` | `src/Pages/Home.vue` |

Every scaffold contains:

```text
frontend/
├── package.json
├── pnpm-workspace.yaml # Svelte only
├── vite.config.ts
├── svelte.config.js  # Svelte only
├── tsconfig.json     # Svelte only
└── src/
    ├── main.ts
    └── Pages/
        └── Home.<framework-extension>
```

The generated Vite config enables the framework plugin, `build.manifest`, and
`src/main.ts` as the Rollup input. Every dependency is pinned to the maintained
example version. Svelte adds `svelte-preprocess` `6.0.5`, a pnpm policy that allows
only its pinned install script, a preprocessing config, and
`verbatimModuleSyntax` TypeScript settings required to build the adapter.

Each generated `Home` imports `Deferred`, renders `greeting` immediately, shows
`Loading stats…`, and then renders the deferred `projects` and `tasks` values. React
always uses `.tsx`.

The command refuses to overwrite an existing `<path>/frontend`. After generation it
prints these next steps:

```text
pnpm --dir frontend install
pnpm --dir frontend build
cargo run
```

It creates frontend files only; add Rust dependencies and the matching immediate-plus-deferred
handler from the [quick start](/docs/getting-started/quick-start).

## `dev`

Defaults:

- frontend directory: `frontend`;
- Vite port: `5173`;
- bind host: `127.0.0.1`.

The command requires `<frontend>/package.json`, then runs:

```text
<selected package manager> run dev --host <host> --port <port>
cargo run
```

The Rust process receives `VITE_DEV_SERVER_URL=http://127.0.0.1:<port>`. Standard input/output/error are inherited. When either child exits, the CLI stops and waits for the peer; a non-success child status makes the command fail.

`dev` currently invokes npm even if the project lockfile uses another package manager. Run your own two-process supervisor when you need pnpm-specific commands, extra Cargo arguments, TLS, a non-loopback Vite host, or restart/watch behavior beyond what Vite provides.

## `check`

`check` defaults to root `.` and frontend directory `frontend` relative to that root. It verifies:

1. the frontend directory and `src/main.ts` exist;
2. `vite.config.ts` contains `src/main.ts`;
3. an existing `dist/.vite/manifest.json` or `dist/manifest.json` is valid JSON;
4. literal `#[inertia(component = "...")]` declarations below the root use safe relative component paths;
5. each declaration is unique and has a matching page below `src/Pages`;
6. frontend page names are unique across `.svelte`, `.tsx`, `.jsx`, and `.vue` files.

Component paths may contain normal slash-separated segments such as `Todos/Index`. Empty segments, parent traversal, absolute paths, and backslashes are rejected.

The checker is intentionally lightweight. It recursively scans `.rs` source text, excluding `target` and `node_modules`; it does not expand macros or query rustc's type information. Only literal component attributes are discovered. It also does not reject frontend pages that have no Rust declaration.

Use `cargo check` and application tests in addition to this command. See [typed pages and props](/docs/guides/typed-pages-and-props) for the compiler-checked boundary.

## `sync`

`sync` compiles filtered `__inertia_typegen_` tests and writes deterministic TypeScript without starting the application. It supports Cargo package, workspace, target, and feature selection; integer and dynamic-page policies; single and managed-module layouts; CI `--check`; and ownership-safe `--clean`. See [Generated TypeScript contracts](/docs/guides/generated-types).
