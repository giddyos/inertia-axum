title: Fallback and strict mode
description: Choose a usable CSR response or a generic 500 when SSR rendering fails.

---

The failure mode is application-wide and applies after an eligible page reaches the renderer.

| Mode | Failed render | Best fit |
| --- | --- | --- |
| Fallback (default) | Returns the same page through the normal CSR mount | Applications that remain usable in the browser |
| Strict | Returns `500 Internal Server Error` | Routes where server-rendered markup is a hard contract |

## Default fallback

```rust title="src/main.rs"
use inertia_axum::Ssr;

let ssr = Ssr::node("dist/ssr/app.js");
```

Fallback covers unavailable backends, load shedding, timeouts, transport and backend errors, malformed or oversized output, and a Vite development `null` response. The failed request is not retried; health becomes degraded and a later successful render restores ready state.

## Strict mode

```rust title="src/main.rs"
use inertia_axum::Ssr;

let ssr = Ssr::node("dist/ssr/app.js").strict();
```

Strict responses never expose backend bodies or JavaScript exceptions. Use `.without_ssr()` when a specific route should remain CSR instead of requiring strict SSR.

The concurrency limiter load-sheds rather than creating an unbounded queue: saturation becomes immediate CSR in fallback mode or an immediate 500 in strict mode. Configure limits and observe health in [Health and operations](/docs/ssr/health-and-operations).
