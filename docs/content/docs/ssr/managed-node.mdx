Run one long-lived Node SSR process alongside Axum without introducing a
separate process manager. The adapter validates the runtime and bundle, waits for
readiness, restarts an exited child, and shuts it down when the last application
handle is dropped.

## Prerequisites

- Enable the `ssr` feature.
- Put Node.js 22 or newer on `PATH`, or configure another executable.
- Produce the SSR bundle described in
  [Frontend build](/docs/ssr/frontend-build).
- Ensure the bundle listens on the same endpoint configured in Rust.

## Start the process

```rust title="src/main.rs"
use inertia_axum::{InertiaApp, Ssr};

let inertia = InertiaApp::vite("frontend")
    .ssr(Ssr::node("dist/ssr/app.js"))
    .start()
    .await?;
```

`.ssr("dist/ssr/app.js")` is equivalent shorthand. The default endpoint is
`http://127.0.0.1:13714`.

With a Vite root, a relative bundle is resolved from that root and Node runs with
the Vite root as its working directory. Without a Vite root, a relative path is
resolved from the Rust process directory and Node runs from the bundle's parent
directory. Absolute bundle paths are preserved.

## Override the runtime or endpoint

```rust title="src/main.rs"
use inertia_axum::Ssr;

let ssr = Ssr::node("dist/ssr/app.js")
    .runtime("/opt/node/bin/node")
    .endpoint("http://127.0.0.1:14000");
```

Only absolute `http://` endpoints are accepted. Changing `.endpoint(...)` changes
where Rust connects; it cannot rewrite a port already compiled into the SSR
bundle. When using a custom port with `@inertiajs/vite`, configure that same port
in the plugin's SSR settings.

## Lifecycle

During `.start().await`, the adapter:

1. Verifies that the bundle exists and is a file.
2. runs the configured executable with `--version` and requires Node major
   version 22 or newer;
3. starts the bundle directly, without a shell;
4. drains stdout and stderr into the `inertia_axum::ssr::node` tracing target;
5. polls `/health` until it succeeds or `startup_timeout` expires.

If the child exits later, health becomes `Unavailable`, then `Starting` while the
supervisor retries with bounded backoff. A successful readiness check restores
`Ready`.

When the last `InertiaApp` handle is dropped, the supervisor requests
`/shutdown`, waits up to the control timeout, and force-kills a process that does
not exit. Keep the application handle alive for as long as the router is serving.

## Production guidance

Managed mode fits a single container or host that contains the Rust binary, Node,
and the bundle. Bind Node to loopback unless another host must reach it. Use an
[external SSR server](/docs/ssr/external-server) when Node runs in another
container, requires clustering, or is already supervised by systemd, Kubernetes,
or another process manager.

Review [Health and operations](/docs/ssr/health-and-operations) before deploying,
and use [Troubleshooting](/docs/ssr/troubleshooting) for startup failures.
