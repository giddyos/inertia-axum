---
title: SSR troubleshooting
description: Diagnose feature, startup, bundle, policy, rendering, capacity, asset, and shutdown failures.
keywords: [SSR errors, missing manifest, Node version, CSR fallback]
---

Start with the visible symptom, confirm the likely cause, then apply the smallest check or fix.

## SSR methods do not exist

**Symptom:** `.ssr(...)`, `.start()`, or route-policy methods do not compile.

**Likely cause:** The non-default `ssr` Cargo feature is disabled or `SsrRouteExt` is not imported.

**Check or fix:** Enable `features = ["ssr"]` and import `inertia_axum::SsrRouteExt as _` for route methods.

## `build()` rejects the configuration

**Symptom:** Startup reports that configured SSR requires asynchronous startup.

**Likely cause:** `.build()` cannot launch or connect an SSR runtime.

**Check or fix:** Replace the final `.build()?` with `.start().await?`.

## Node is missing or unsupported

**Symptom:** Managed startup cannot execute Node or rejects its version.

**Likely cause:** Node is absent from `PATH`, the configured executable is invalid, or its major version is below 22.

**Check or fix:** Run `node --version`, install Node 22+, or pass the correct absolute executable through `.runtime(...)`.

## The bundle cannot be found

**Symptom:** Managed startup reports a missing or non-file bundle.

**Likely cause:** The SSR build was skipped or the path was resolved from a different Vite/process root.

**Check or fix:** Run the two-output [frontend build](/docs/ssr/frontend-build) and compare the emitted file with `.ssr(...)`.

## Production Vite reports a missing manifest

**Symptom:** Production startup fails before contacting the SSR backend.

**Likely cause:** The client build or configured entry/build directory does not match the manifest.

**Check or fix:** Verify the canonical [client-manifest contract](/docs/ssr/frontend-build#connect-both-outputs). SSR does not replace the browser build.

## Startup times out waiting for health

**Symptom:** `.start().await` reaches `startup_timeout`.

**Likely cause:** `/health` is unreachable, the endpoint differs between Rust and the bundle, or the renderer failed during boot.

**Check or fix:** Call the configured health URL from the Rust host, inspect renderer logs, and align both endpoint settings before increasing the timeout.

## A page renders as CSR

**Symptom:** The page is usable but contains no server-rendered marker.

**Likely cause:** Fallback handled a render failure, the route opted out, opt-in mode lacks `.with_ssr()`, or `ssr_when` returned false.

**Check or fix:** Inspect `ssr_health()` and tracing, then check [route policies](/docs/ssr/route-policies). Temporarily use strict mode only to make a hidden backend failure visible as a generic 500.

## Strict mode returns 500

**Symptom:** An eligible page returns `500 Internal Server Error`.

**Likely cause:** The same backend, timeout, overload, or response validation failure that fallback would convert to CSR occurred under `.strict()`.

**Check or fix:** Inspect health and server-side tracing; raw backend details are intentionally not sent to the browser.

## Renders are overloaded or slow

**Symptom:** Requests fall back or fail immediately under load, or hit the render timeout.

**Likely cause:** `max_concurrency` is saturated or renderer CPU/latency exceeds configured capacity.

**Check or fix:** Measure render latency, CPU, memory, and fallback rate before tuning the limits in [Health and operations](/docs/ssr/health-and-operations).

## SSR output is rejected

**Symptom:** Health records an invalid, render-error, or oversized response.

**Likely cause:** `/render` returned a non-success status, malformed JSON, a wrong `head`/`body` shape, or more than `max_response_bytes`.

**Check or fix:** Capture the renderer response server-side and compare it with the external response contract; raise the size limit only for measured legitimate documents.

## Head or mount markup appears twice

**Symptom:** Titles, head tags, the app root, or initial page data are duplicated.

**Likely cause:** A custom root serializes page/SSR fragments itself in addition to the adapter-provided fragments.

**Check or fix:** Emit each marker once and let the selected `RootView` place the trusted assets, head, and mount values.

## Browser assets return 404

**Symptom:** SSR HTML arrives, but hydration scripts or styles fail to load.

**Likely cause:** `.entry(...)`, `.build_dir(...)`, `.public_path(...)`, or proxy routing disagrees with the client manifest.

**Check or fix:** Inspect the manifest entry and emitted URLs, then verify Axum or the proxy serves the configured public prefix.

## Managed Node does not release its port

**Symptom:** Restarting Rust reports that the renderer endpoint is already in use.

**Likely cause:** An `InertiaApp` handle still owns the supervisor or shutdown exceeded the control deadline.

**Check or fix:** Drop every application clone during normal shutdown, inspect `/shutdown` handling, and allow the supervisor to force-kill after its bounded wait.
