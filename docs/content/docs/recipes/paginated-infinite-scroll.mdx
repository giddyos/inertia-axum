title: Add paginated infinite scroll
description: Normalize an Axum paginator for Inertia v3 and render it with InfiniteScroll.

---

Return a `ScrollPage` so the Inertia v3 client can load adjacent pages, merge items,
and keep the query string synchronized with the visible page.

## 1. Read the page query

```rust
use axum::extract::Query;
use serde::Deserialize;

#[derive(Deserialize)]
struct Pagination {
    #[serde(default = "first_page")]
    page: u64,
}

fn first_page() -> u64 { 1 }
```

## 2. Return normalized data

```rust
use inertia_axum::{ScrollPage, prelude::*};

async fn index(Query(query): Query<Pagination>) -> DynamicPage {
    let result = load_page(query.page).await;
    let mut page = ScrollPage::new(result.items, query.page)
        .page_name("page");

    if query.page > 1 {
        page = page.previous(query.page - 1);
    }
    if result.has_more {
        page = page.next(query.page + 1);
    }

    page!("Activity/Index", {
        activity: scroll(page).match_on("id"),
    })
}
# struct ActivityPage { items: Vec<serde_json::Value>, has_more: bool }
# async fn load_page(_: u64) -> ActivityPage { ActivityPage { items: vec![], has_more: false } }
```

The prop value is `{ data: [...] }`. `scrollProps.activity` records the page query
name, current page, and available previous/next pages. `match_on("id")` lets the
client replace an existing item with the same identity.

For two scroll regions, give each a different `.page_name(...)` so their query
parameters do not collide.

## 3. Render with the client component

```vue
<script setup>
import { InfiniteScroll } from '@inertiajs/vue3'
defineProps({ activity: Object })
</script>

<template>
  <InfiniteScroll data="activity">
    <article v-for="item in activity.data" :key="item.id">
      {{ item.title }}
    </article>
  </InfiniteScroll>
</template>
```

Svelte and React export `InfiniteScroll` from their framework package with the same
`data` key. The component can load in both directions, preserve the original URL,
use a custom scroll container, or expose manual controls.

## 4. Reset when filters change

```ts
router.reload({
  data: { status },
  only: ['activity'],
  reset: ['activity'],
})
```

Without `reset`, results for the new filter would merge with the old list. The server
removes scroll merge metadata for the reset response while still returning the prop.

Full page visits replace values regardless of scroll metadata. Test next/previous
metadata and reset behavior; the advanced `incident-board` example exercises scroll
and merge policies. See [Examples](/docs/reference/examples).
