The default `vite` feature turns a frontend directory into development script tags
or production manifest assets. It also derives the Inertia asset version and mounts
the compiled directory as an Axum static service.

## Use the conventions

```rust title="src/main.rs"
use inertia_axum::InertiaApp;

let inertia = InertiaApp::vite("frontend").build()?;
```

The conventional entry is `src/main.ts`, the build directory is `dist`, and the
public URL prefix is `/build`.

## Run development mode

Start Vite, then export its absolute URL for the Axum process:

```bash
VITE_DEV_SERVER_URL=http://127.0.0.1:5173 cargo run
```

The root document receives Vite's `/@vite/client` and configured entry scripts. No
manifest, asset version, or Axum static mount is used in this mode. The URL must be
an absolute `http://` or `https://` URL.

`cargo inertia dev` can coordinate Vite and Cargo for its generated npm projects;
read its [documented limits](/docs/guides/cargo-inertia#run-the-development-processes).

## Build for production

Configure Vite to emit a client manifest:

```js title="frontend/vite.config.js"
import inertia from '@inertiajs/vite'
import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [inertia()],
  build: {
    manifest: true,
    rolldownOptions: {
      input: 'src/main.ts',
    },
  },
})
```

Run `pnpm build` before starting Axum. Without a development URL, startup reads
`frontend/dist/.vite/manifest.json`. Missing, malformed, or incomplete manifests are
configuration errors and stop the application before it serves requests.

## Override the layout

The production browser examples place output outside the frontend root:

```rust title="src/main.rs"
let inertia = InertiaApp::vite(frontend_root)
    .entry("src/app.js")
    .build_dir("../public/build")
    .public_path("/public/build")
    .build()?;
```

`public_path` must be an absolute, HTML-safe URL path. Hashing, version-conflict
behavior, builder precedence, and custom providers remain in [Configuration](/docs/reference/configuration)
and [Custom asset provider](/docs/advanced/custom-asset-provider).

Production Vite needs the **client manifest even when SSR is enabled**. The SSR wire
protocol does not consume it, but the initial root still needs client assets and the
adapter uses it for versioning and static serving.
